根据上传的 AI Grand Prix Virtual Qualifier 技术规格文件，比赛核心不是“写一个 AI 模型”本身，而是做一个**无人机自主飞行系统**：用前视相机和 MAVLink 遥测，在仿真环境中自动穿过起点门、中间门和终点门，并且提交计时飞行时不能有人为干预。

## 1. 比赛目标

Round One 的目标是验证参赛软件能否成功导航完整赛道。赛道结构包括：

| 项目   | 内容                                         |
| ---- | ------------------------------------------ |
| 起点   | start gate                                 |
| 中间赛道 | intermediate gates / sequential race gates |
| 终点   | finish gate                                |
| 最长时间 | 8 分钟                                       |
| 提交要求 | 计时飞行过程中禁止人为干预，否则直接取消资格                     |

文档没有给出明确评分公式，比如是否按总时间排名、是否有碰撞惩罚、漏门惩罚、重置惩罚等。因此目前只能确定：**必须自主完成赛道；高分策略应优先保证稳定完成，再优化速度**。

## 2. 硬性限制

### 2.1 不能人工接管

提交的 timed run 飞行过程中，任何 human interaction 都会导致 immediate disqualification。这意味着所有感知、规划、控制、纠错都必须在软件中自动完成。

### 2.2 运行环境限制

文档说明仿真器运行在 Windows 11，标准 PC，GPU 显存约 8GB；当前不支持 Linux。参赛者可以使用 Python，文档提到 Python 3.14.2 可正常运行，也允许其他环境。

工程含义：不要把主开发环境只放在 Linux 上。需要尽早在 Windows 11 上验证依赖、网络、MAVLink、图像流解码和控制频率。

### 2.3 控制频率限制

仿真物理频率是 120 Hz，但控制命令频率必须小于 100 Hz，heartbeat 最低 2 Hz。

建议控制频率设置在 **50–80 Hz**，视觉处理 30 Hz，控制层在两帧图像之间做状态预测，不要每收到图像才发一次控制。

### 2.4 没有 GPS 和全局位置

仿真器内部使用局部笛卡尔坐标，但文档明确说没有 GPS 仿真，也不暴露 absolute global position。

这很关键。不能依赖“知道自己在赛道全局坐标中的位置”。系统需要主要依赖：

1. 前视相机识别 gate；
2. IMU、姿态、速度等遥测；
3. 视觉相对定位；
4. 必要时用已知赛道的确定性做前馈路径。

## 3. 提供的技术资料

### 3.1 仿真环境

仿真环境包括起点门、顺序赛门、终点门、垂直和水平障碍、边界、地形与环境结构。赛道几何、物理参数和环境条件对所有参赛者一致，并且是 deterministic。

这意味着可以做大量重复测试，利用同一赛道的确定性调参、优化轨迹、记录失败案例。

### 3.2 飞行器尺寸

无人机尺寸：

| 尺寸 |     数值 |
| -- | -----: |
| 宽  | 280 mm |
| 长  | 280 mm |
| 高  | 160 mm |

Gate 外边界为 2700 × 2700 × 260 mm，内孔为 1500 × 1500 × 260 mm。

由此可估算安全余量：

| 方向 |    内孔尺寸 |   机体尺寸 |   单侧理论余量 |
| -- | ------: | -----: | -------: |
| 横向 | 1500 mm | 280 mm | 约 610 mm |
| 垂直 | 1500 mm | 160 mm | 约 670 mm |

理论余量不小，但高速飞行时姿态倾斜、控制延迟、视觉误差、碰撞模型都会吃掉余量。实际策略不应贴边飞，建议让 gate center 误差控制在 **0.3–0.4 m 以内**，高速段也尽量不要超过 0.5 m。

### 3.3 坐标系与相机

MAVLink 坐标 convention 是 NED：

| Frame               | 含义                      |
| ------------------- | ----------------------- |
| MAV_FRAME_LOCAL_NED | 原点为地面固定点，通常是无人机 arm 的位置 |
| MAV_FRAME_BODY_NED  | 原点为机体，X 向前，Y 向右，Z 向下    |

相机和机体同原点，相机相对机体向上倾斜 20°。IMU 到机体是 identity map。相机是 pinhole model，无畸变，分辨率 640 × 360，主点 [320, 180]，fx = fy = 320。

注意：文档写了 “VFoV = 90°”，但 fx = fy = 320 且分辨率 640 × 360 时，标准 pinhole 计算得到 horizontal FOV 约 90°，vertical FOV 约 58.7°。这里可能存在“VFoV/HFoV 标注不一致”。实现投影、PnP、视觉伺服前，必须用仿真图像实测验证内参，不要盲信单一字段。

### 3.4 通信接口

通信通过 MAVLink 2 / MAVSDK-compatible interface，传输使用 UDP。支持消息包括：

| 消息                            | 方向                 | 用途        |
| ----------------------------- | ------------------ | --------- |
| HEARTBEAT                     | Simulator → Client | 连接状态      |
| ATTITUDE                      | Simulator → Client | 姿态        |
| HIGHRES_IMU                   | Simulator → Client | 车辆状态 / 测量 |
| SET_POSITION_TARGET_LOCAL_NED | Client → Simulator | 控制接口      |
| SET_ATTITUDE_TARGET           | Client → Simulator | 控制接口      |
| TIMESYNC                      | Simulator → Client | 时间同步      |



文档没有说明 `SET_POSITION_TARGET_LOCAL_NED` 中哪些字段或 type mask 被仿真器实际使用，也没有说明 `SET_ATTITUDE_TARGET` 的 thrust/attitude/yaw-rate 具体语义。因此开发时要尽早做接口实验：分别发送速度、位置、加速度、姿态、推力命令，记录无人机响应，确认真实控制通道。

### 3.5 图像流

相机图像流频率为 30 Hz，分辨率 640 × 360，UDP 默认端口 5600。每张 JPEG 图像被拆成多个 packet，每个 packet 有 24 字节 metadata header，包含 frame_id、chunk_id、total_chunks、jpeg_size、payload_size 和 sim_time_ns。

工程重点：

1. 必须按 frame_id 重组 packet；
2. 必须能处理乱序、丢包、重复包；
3. incomplete frame 要丢弃，不要把坏图送进感知模型；
4. 用 sim_time_ns 对齐图像和遥测；
5. 记录图像延迟，否则高速飞行会明显偏门。

### 3.6.1 启动和连接流程

当前模板的实际启动链路是：

```text
启动 AIGP_3364/FlightSim.exe
→ 进入仿真器并让它开始发送 MAVLink / vision 数据
→ python PyAIPilotExample/main.py
→ setup_components(...)
→ mavutil.mavlink_connection("udpin:127.0.0.1:14550")
→ wait_heartbeat()
→ 启动 MAVLinkRX 接收线程
→ 启动 TimeSync 请求线程
→ 启动 VisionRX 图像接收线程
→ 创建 Controller
→ controller.arm()
→ while True: controller.update()
```

也就是说，Python 客户端默认在本机监听 MAVLink UDP `127.0.0.1:14550`，图像接收器默认绑定 `0.0.0.0:5600`。如果仿真器和 Python 不在同一台机器，需要同时修改 `main.py` 里的 MAVLink 地址，以及确认仿真器的 UDP 输出目标。

### 3.6.2 代码中的实际数据流

MAVLink 状态流：

```text
Simulator
→ UDP 14550
→ setup.py / mavutil.mavlink_connection(...)
→ mavlink_rx.py / recv_match(blocking=False)
→ on_heartbeat / on_attitude / on_odometry / on_highres_imu / on_collision / ...
→ shared_data 或日志系统
→ controller.py
```

图像流：

```text
Simulator
→ UDP 5600
→ vision_rx.py / sock.recvfrom(...)
→ 24 字节 header: frame_id, chunk_id, total_chunks, jpeg_size, payload_size, sim_time_ns
→ 按 frame_id 和 chunk_id 重组 JPEG
→ cv2.imdecode(...)
→ process_frame(frame_id, image)
→ gate 检测 / 位姿估计
→ shared_data 或日志系统
→ controller.py
```

控制命令流：

```text
controller.py
→ arm: MAV_CMD_COMPONENT_ARM_DISARM
→ reset: 自定义 MAVLINK_CMD_SIM_RESET = 31000
→ set_actuator_control_target_send(...)
→ set_attitude_target_send(...)
→ set_position_target_local_ned_send(...)
→ Simulator
```

代码里已经提供了三类控制接口：

| 函数 | MAVLink 消息 | 当前含义 |
| ---- | ------------ | -------- |
| `update_motor_control()` | `SET_ACTUATOR_CONTROL_TARGET` | 直接发送 8 路 actuator/motor 数组 |
| `update_attitude_flight_control()` | `SET_ATTITUDE_TARGET` | 发送 roll/pitch/yaw rate 和 thrust |
| `update_position_flight_control()` | `SET_POSITION_TARGET_LOCAL_NED` | 示例发送 NED 速度目标 |

当前 `Controller.update()` 默认从 `KeyboardControlConsole` 读取 `FlightCommand`，再调用 `update_attitude_flight_control()` 发送姿态角速度和推力目标。这仍然是人工键盘控制样例，不是能提交 timed run 的自主 baseline。

### 3.6.3 MAVLink 解析内容和可利用信息

`mavlink_rx.py` 目前能识别这些消息类型：

| 消息 | 代码处理函数 | 可用信息 |
| ---- | ------------ | -------- |
| `HEARTBEAT` | `on_heartbeat()` | 是否 armed |
| `TIMESYNC` | `on_timesync()` | 请求/响应时间戳 |
| `ATTITUDE` | `on_attitude()` | roll、pitch、yaw、角速度 |
| `LOCAL_POSITION_NED` | `on_local_position_ned()` | NED 位置和速度 |
| `ODOMETRY` | `on_odometry()` | 位置、四元数、速度、角速度、reset_count |
| `HIGHRES_IMU` | `on_highres_imu()` | 加速度和陀螺仪 |
| `ACTUATOR_OUTPUT_STATUS` | `on_actuator_output_status()` | 4 个电机输出 |
| `COLLISION` | `on_collision()` | gate / environment 碰撞 ID、威胁等级、冲量 |
| `ENCAPSULATED_DATA` | `on_encapsulated_data()` | race status 或 track data |
| `DATA_TRANSMISSION_HANDSHAKE` | 接收循环内处理 | track data 分片数量 |

其中 `ENCAPSULATED_DATA` 有两个自定义 payload：

| data_type | 含义 | 字段 |
| --------- | ---- | ---- |
| `1` | race status | sim_boot_time_ms、race_start_boot_time_ms、race_finish_time_ns、active_gate_index、last_gate_race_time |
| `2` | track info | gate_id、gate 在 NED 坐标中的位置、四元数朝向、width、height |

这说明代码模板实际上暴露了比技术规格摘要更多的信息：不仅有相机和基础遥测，还有当前 active gate index、赛道 gate 几何、碰撞事件和 actuator output。这些信息能显著降低开发难度，但是否允许在正式提交中依赖 track info 做赛道特化，需要再确认主办方规则。

### 3.6.4 Python 接收侧文件的输入输出

这里的“输入输出”指 Python 文件实际处理的数据流，不是比赛规则层面的输入输出。当前模板不是函数式 pipeline，大多数文件通过线程、socket、MAVLink 连接和 `shared_data` 产生副作用。

依据说明：

| 内容 | 依据 | 是否实测模拟器输出 |
| ---- | ---- | ---------------- |
| MAVLink 消息类型、读取字段、handler 函数 | `PyAIPilotExample/mavlink_rx.py` 源码 | 否，来自代码 |
| 图像 UDP header 格式 `"<IHHIIQ"` | `PyAIPilotExample/vision_rx.py` 源码 | 否，来自代码 |
| race status / track info 的 `struct.unpack_from(...)` 格式 | `PyAIPilotExample/mavlink_rx.py` 源码 | 否，来自代码 |
| `shared_data[...]` 输出结构 | 根据现有代码缺口给出的建议格式 | 否，是建议 |
| 后面的具体数字例子 | 为说明格式构造的样例值 | 否，不是抓包数据 |

接收侧 Python 文件 I/O 总览：

| Python 文件 | 输入 | 处理 | 当前输出 | 应补输出 |
| ---- | ---- | ---- | ---- | ---- |
| `main.py` | 常量 `SIM_SERVER_UDP_IP`、`SIM_SERVER_UDP_PORT`；空 `shared_data`；本机启动时间 `system_boot_ms` | 调用 `setup_components()`，启动各组件，arm 后进入控制循环 | 组件对象、控制循环、副作用命令发送 | 配置文件读取、日志路径、运行状态、退出码 |
| `setup.py` | `shared_data`、`system_boot_ms`、MAVLink IP/port | 建立 MAVLink UDP 连接，等待 heartbeat，创建 MAVLink RX、TimeSync、VisionRX、Controller | 返回 `dict`：`vision_rx`、`mavlink_rx`、`ts_loop`、`sim_conn`、`controller` | 加入共享锁、logger、配置对象 |
| `mavlink_rx.py` | `mavlink_connection.recv_match()` 收到的 MAVLink 消息 | 按消息类型分发到 `on_heartbeat()`、`on_attitude()`、`on_track_data()` 等函数 | 当前大多只生成局部变量；track data 分片暂存在 `track_chunks` | 写入 `shared_data["telemetry"]`、`shared_data["race"]`、`shared_data["track"]`、`shared_data["events"]` |
| `vision_rx.py` | UDP `5600` 收到的图像分片 packet | 解析 24 字节 header，按 `frame_id/chunk_id` 重组 JPEG，用 OpenCV 解码 | 调用空的 `process_frame(frame_id, image)`；当前不保存结果 | 写入 `shared_data["vision"]`，包括 `frame_id`、`sim_time_ns`、图像、gate 检测结果 |
| `timesync.py` | 本机 `time.time_ns()`；MAVLink 连接 | 以 10 Hz 发送 TIMESYNC request | 向模拟器发送 `TIMESYNC` 消息 | 配合 `mavlink_rx.py/on_timesync()` 计算并保存时钟偏移 |

从代码模板看，模拟器至少会通过两条通道给客户端数据：MAVLink UDP `14550` 和视觉 UDP `5600`。MAVLink 由 `mavlink_rx.py` 用 `recv_match(blocking=False)` 接收；图像由 `vision_rx.py` 用 UDP socket 绑定 `0.0.0.0:5600` 接收。

`mavlink_rx.py` 处理的输入输出更具体地说是：

| 处理函数 | 输入消息 | 读取字段 | 当前 Python 输出 |
| ---- | ---- | ---- | ---- |
| `on_heartbeat(msg)` | `HEARTBEAT` | `msg.base_mode` | 局部变量 `armed`，未保存 |
| `on_timesync(msg)` | `TIMESYNC` | `msg.ts1`、`msg.tc1` | 局部变量 `request_time`、`response_time`，未保存 |
| `on_attitude(msg)` | `ATTITUDE` | `time_boot_ms`、roll/pitch/yaw、rollspeed/pitchspeed/yawspeed | 局部姿态变量，未保存 |
| `on_local_position_ned(msg)` | `LOCAL_POSITION_NED` | `x/y/z`、`vx/vy/vz`、`time_boot_ms` | 局部位置速度变量，未保存 |
| `on_odometry(msg)` | `ODOMETRY` | `x/y/z`、`q`、`vx/vy/vz`、角速度、`time_usec`、`reset_counter` | 局部里程计变量，未保存 |
| `on_highres_imu(msg)` | `HIGHRES_IMU` | `xacc/yacc/zacc`、`xgyro/ygyro/zgyro`、`time_usec` | 局部 IMU 变量，未保存 |
| `on_actuator_output_status(msg)` | `ACTUATOR_OUTPUT_STATUS` | `actuator[0:4]`、`time_usec` | 局部电机反馈变量，未保存 |
| `on_collision(msg)` | `COLLISION` | `id`、`threat_level`、`horizontal_minimum_delta` | 局部碰撞变量，未保存 |
| `on_race_status(msg)` | `ENCAPSULATED_DATA` type `1` | 自定义 race status payload | 局部比赛状态变量，未保存 |
| `on_track_data_packet(msg)` | `ENCAPSULATED_DATA` type `2` 分片 | `data_type`、`transfer_id`、`msg.seqnr`、payload | 分片写入 `self.track_chunks`，收齐后调用 `on_track_data()` |
| `on_track_data(payload)` | 拼好的 track payload | `num_gates` 和每个 gate 的位置/朝向/宽高 | 局部 gate 变量，未保存 |

也就是说，`mavlink_rx.py` 的输入已经很完整，但输出目前基本没有落到共享状态里。下一步应该把每个 `on_*()` 的局部变量变成结构化字典，写入 `self.data`，也就是 `shared_data`。

建议输出格式如下：

```python
shared_data["telemetry"]["attitude"] = {
    "time_boot_ms": time_boot_ms,
    "roll": roll,
    "pitch": pitch,
    "yaw": yaw,
    "rollspeed": roll_speed,
    "pitchspeed": pitch_speed,
    "yawspeed": yaw_speed,
}
```

格式示例，数值是构造样例，不是实测抓包：

```python
# mavlink_rx.py 输入：一个 ATTITUDE MAVLink 消息
msg.time_boot_ms = 15320
msg.roll = 0.03
msg.pitch = -0.08
msg.yaw = 1.57
msg.rollspeed = 0.10
msg.pitchspeed = -0.20
msg.yawspeed = 0.05

# mavlink_rx.py 应输出到 shared_data
shared_data["telemetry"]["attitude"] = {
    "time_boot_ms": 15320,
    "roll": 0.03,
    "pitch": -0.08,
    "yaw": 1.57,
    "rollspeed": 0.10,
    "pitchspeed": -0.20,
    "yawspeed": 0.05,
}
```

`vision_rx.py` 处理的输入输出更具体地说是：

| 阶段 | 输入 | 处理 | 输出 |
| ---- | ---- | ---- | ---- |
| UDP 接收 | `sock.recvfrom(65536)` 返回的 `packet` | 拆成 `header` 和 `payload` | 原始分片 |
| header 解析 | `packet[:24]` | `struct.unpack("<IHHIIQ", header)` | `frame_id`、`chunk_id`、`total_chunks`、`jpeg_size`、`payload_size`、`sim_time_ns` |
| 分片缓存 | `frame_id`、`chunk_id`、`payload` | 写入 `frames[frame_id]["chunks"][chunk_id]` | 内存中的分片表 |
| JPEG 重组 | 同一帧的全部 chunk | 按 `0..total_chunks-1` 拼成 `jpeg_bytes` | 完整 JPEG bytes |
| 图像解码 | `jpeg_bytes` | `np.frombuffer()` + `cv2.imdecode(..., cv2.IMREAD_COLOR)` | OpenCV BGR 图像 `image` |
| 感知处理 | `frame_id`、`image` | 当前 `process_frame()` 是空函数 | 当前无输出；应输出 gate detection |

建议 `vision_rx.py` 的最终输出格式如下：

```python
shared_data["vision"] = {
    "frame_id": frame_id,
    "sim_time_ns": sim_time_ns,
    "image_shape": image.shape,
    "gate_detection": {
        "bbox_xyxy": [x1, y1, x2, y2],
        "center_px": [cx, cy],
        "confidence": confidence,
        "corners_px": [[x0, y0], [x1, y1], [x2, y2], [x3, y3]],
    },
}
```

格式示例，数值是构造样例，不是实测抓包：

```python
# vision_rx.py 输入：某个 UDP 图像分片 header 解包结果
frame_id = 42
chunk_id = 0
total_chunks = 3
jpeg_size = 84211
payload_size = 30000
sim_time_ns = 123456789000

# 收齐 3 个 chunk 并解码后，假设 gate 检测结果如下
shared_data["vision"] = {
    "frame_id": 42,
    "sim_time_ns": 123456789000,
    "image_shape": (360, 640, 3),
    "gate_detection": {
        "bbox_xyxy": [250, 90, 390, 230],
        "center_px": [320, 160],
        "confidence": 0.92,
        "corners_px": [[255, 95], [385, 92], [390, 228], [250, 230]],
    },
}
```

图像 UDP 包的具体格式在 `vision_rx.py` 中写死为 little-endian：

```python
header_format = "<IHHIIQ"
```

24 字节 header 的字段顺序是：

| 偏移 | 字段 | 类型 | 含义 |
| ---: | ---- | ---- | ---- |
| 0 | `frame_id` | `uint32` | 图像帧编号 |
| 4 | `chunk_id` | `uint16` | 当前分片编号，从 `0` 开始 |
| 6 | `total_chunks` | `uint16` | 这一帧 JPEG 总分片数 |
| 8 | `jpeg_size` | `uint32` | 完整 JPEG 字节数 |
| 12 | `payload_size` | `uint32` | 当前 UDP 包中 JPEG 分片字节数 |
| 16 | `sim_time_ns` | `uint64` | 该图像帧在模拟器服务器上的时间戳，单位 ns |

header 后面紧跟 `payload_size` 字节的 JPEG 分片。接收端按 `frame_id` 分组、按 `chunk_id = 0..total_chunks-1` 拼接，得到完整 `jpeg_bytes` 后用：

```python
img_array = np.frombuffer(jpeg_bytes, dtype=np.uint8)
image = cv2.imdecode(img_array, cv2.IMREAD_COLOR)
```

因此 `process_frame()` 拿到的是 OpenCV BGR 图像，不是 RGB 图像。当前函数签名是 `process_frame(frame_id, image)`，没有把 `sim_time_ns` 传下去；后续应改成 `process_frame(frame_id, sim_time_ns, image)` 或把时间戳写入 `shared_data`。

比赛状态 payload 的具体格式如下：

```python
struct.unpack_from("<BQqqIq", raw_payload)
```

字段顺序是：

| 字段 | 类型 | 含义 |
| ---- | ---- | ---- |
| `data_type` | `uint8` | 固定为 `1`，表示 race status |
| `sim_boot_time_ms` | `uint64` | 模拟器启动后的服务器时间，单位 ms |
| `race_start_boot_time_ms` | `int64` | 比赛开始时的 boot time；小于 `0` 表示未开始 |
| `race_finish_time_ns` | `int64` | 比赛结束时相对模拟器启动的时间戳，单位 ns；小于 `0` 表示仍在进行 |
| `active_gate_index` | `uint32` | 当前目标 gate 的索引 |
| `last_gate_race_time` | `int64` | 最近一次过门对应的比赛时间；代码注释写的是 seconds |

赛道信息不是一个 MAVLink 消息直接装完，而是先用 `DATA_TRANSMISSION_HANDSHAKE` 建立分片上下文，再用多条 `ENCAPSULATED_DATA` 拼接：

1. `DATA_TRANSMISSION_HANDSHAKE.width` 被模板当作 `transfer_id`。
2. `DATA_TRANSMISSION_HANDSHAKE.packets` 表示这个 transfer 需要多少个 track data 分片。
3. 每个 `ENCAPSULATED_DATA` 的前 3 字节用 `struct.unpack_from("<BH", raw_payload)` 解析为 `data_type` 和 `transfer_id`。
4. `data_type = 2` 表示 track info；剩余字节按 `msg.seqnr` 存入 `track_chunks[transfer_id]`。
5. 收齐 `packets` 个分片后，按 `seqnr = 0..packets-1` 顺序拼接成 `full_payload`。

拼好的 track payload 格式是：

```python
num_gates, = struct.unpack_from("<H", payload)
gate = struct.unpack_from("<Hfffffffff", payload)
```

其中 payload 先是 2 字节 `num_gates`，后面每个 gate 固定 38 字节：

| 字段 | 类型 | 含义 |
| ---- | ---- | ---- |
| `gate_id` | `uint16` | gate 编号 |
| `position_ned_x` | `float32` | gate 中心 NED X，单位 m |
| `position_ned_y` | `float32` | gate 中心 NED Y，单位 m |
| `position_ned_z` | `float32` | gate 中心 NED Z，单位 m |
| `orientation_ned_w` | `float32` | gate 朝向四元数 w |
| `orientation_ned_x` | `float32` | gate 朝向四元数 x |
| `orientation_ned_y` | `float32` | gate 朝向四元数 y |
| `orientation_ned_z` | `float32` | gate 朝向四元数 z |
| `width` | `float32` | gate 宽度，单位 m |
| `height` | `float32` | gate 高度，单位 m |

`ODOMETRY.q` 和 track info 里的 gate orientation 都按四元数 `w, x, y, z` 存储；`mavlink_rx.py` 里只是为了本地变量命名，把 `msg.q[1]`、`msg.q[2]`、`msg.q[3]`、`msg.q[0]` 映射成 `qx, qy, qz, qw`。

这里最重要的结论是：**代码模板能拿到所有 gate 的 NED 坐标、朝向和尺寸，也能拿到当前 active gate index**。这意味着我们不仅能做纯视觉伺服，还可以做“赛道信息 + 视觉校正”的混合方案。

但要注意三点：

1. `mavlink_rx.py` 目前只是把字段 unpack 到局部变量，并没有写入 `shared_data`，所以控制器暂时用不到这些信息。
2. `vision_rx.py` 解析到了 `sim_time_ns`，但当前没有向后传递；高速飞行时必须保存这个时间戳，否则无法做图像延迟补偿。
3. `LOCAL_POSITION_NED`、`ODOMETRY` 和 track info 看起来是仿真器真值级别的信息；技术规格又强调没有 GPS / absolute global position。因此正式比赛中是否允许依赖这些信息，需要确认规则。工程上可以先用于开发、日志和验证，再逐步降低对绝对位置的依赖。

建议把可观测状态统一整理成这样的 `shared_data` 结构：

```python
shared_data = {
    "telemetry": {
        "attitude": {...},
        "local_position_ned": {...},
        "odometry": {...},
        "imu": {...},
        "actuator_output": {...},
    },
    "race": {
        "active_gate_index": None,
        "race_start_boot_time_ms": None,
        "race_finish_time_ns": None,
        "last_gate_race_time": None,
    },
    "track": {
        "gates": [],
    },
    "vision": {
        "frame_id": None,
        "sim_time_ns": None,
        "image": None,
        "gate_detection": None,
    },
    "events": {
        "last_collision": None,
    },
}
```

### 3.6.5 Python 控制侧文件的输入输出

控制侧的 Python 文件负责把“控制输入”变成 MAVLink 命令输出到模拟器。当前模板的控制输入来自键盘；正式自动驾驶版本应把输入改成 `shared_data` 里的视觉、姿态、速度、race status 和 track info。

依据说明：

| 内容 | 依据 | 是否实测模拟器输出 |
| ---- | ---- | ---------------- |
| `FlightCommand` 字段 | `PyAIPilotExample/flight_command.py` 源码 | 不涉及模拟器输出 |
| 键盘到控制量的映射 | `PyAIPilotExample/keyboard_control.py` 源码 | 不涉及模拟器输出 |
| arm/reset/attitude/position/timesync 调用格式 | `PyAIPilotExample/controller.py`、`timesync.py` 源码 | 否，来自代码 |
| `AutonomyCommandSource` | 为说明自动驾驶替换方式给出的建议结构 | 否，是建议 |
| 后面的控制数字例子 | 为说明格式构造的样例值 | 否，不是实测飞行数据 |

控制侧 Python 文件 I/O 总览：

| Python 文件 | 输入 | 处理 | 当前输出 | 自动驾驶版本应输出 |
| ---- | ---- | ---- | ---- | ---- |
| `keyboard_control.py` | 键盘状态：`A/D/W/S/Q/E/Space/Esc` | 读取按键并映射成角速度和推力 | `FlightCommand` 对象 | 正式 timed run 不应使用 |
| `flight_command.py` | 无实时输入；定义数据结构 | 定义 `FlightCommand` dataclass | 统一的控制命令格式 | 可继续作为自动驾驶控制器输出格式 |
| `controller.py` | `sim_conn`、`shared_data`、`system_boot_ms`、`command_source` | 每帧调用 `command_source.read_command()`，再发送 MAVLink 控制命令 | 默认输出 `SET_ATTITUDE_TARGET`；`update()` 返回是否继续运行 | 根据 `shared_data` 自动计算 `FlightCommand` 并发送 |
| `timesync.py` | 本机时间、MAVLink 连接 | 10 Hz 构造 TIMESYNC request | `TIMESYNC` MAVLink 消息 | 同时写入同步偏移供日志/延迟补偿使用 |
| `main.py` | 组件对象、用户中断 | arm，无限循环调用 `controller.update()`，退出时 join 线程 | `MAV_CMD_COMPONENT_ARM_DISARM` 和持续控制命令 | 批量测试、自动 reset、日志 flush |

控制输入接口本身不是 MAVLink，而是 `Controller` 依赖的 `command_source` 对象。这个对象只需要满足一个最小接口：

```python
class CommandSource:
    def read_command(self) -> FlightCommand:
        ...

    def close(self) -> None:
        ...
```

其中 `close()` 是可选的，`Controller.shutdown()` 会用 `getattr()` 检查是否存在。真正必须实现的是 `read_command()`，它每个控制周期返回一个 `FlightCommand`。

当前 Python 模板通过 MAVLink 向模拟器发送控制命令。控制入口主要在 `controller.py`，当前默认流程是：

```text
main.py
→ setup_components(...)
→ KeyboardControlConsole()
→ Controller(...)
→ controller.arm()
→ while is_running: controller.update()
→ command_source.read_command()
→ update_attitude_flight_control(...)
→ SET_ATTITUDE_TARGET
```

也就是说，现有模板默认是键盘控制样例，不是自主控制。键盘层生成的命令对象格式来自 `flight_command.py`：

```python
@dataclass(frozen=True)
class FlightCommand:
    roll_rate: float
    pitch_rate: float
    yaw_rate: float
    thrust: float
    exit_requested: bool = False
```

`KeyboardControlConsole` 的按键到命令映射如下：

| 按键 | 字段变化 | 当前幅值 |
| ---- | -------- | -------- |
| `A` / `D` | `roll_rate` 负 / 正 | `1.0 rad/s` |
| `W` / `S` | `pitch_rate` 负 / 正 | `1.0 rad/s` |
| `Q` / `E` | `yaw_rate` 负 / 正 | `1.0 rad/s` |
| `Space` | `thrust = 0.6`，否则 `0.0` | normalized thrust |
| `Esc` | `exit_requested = True` | 退出控制循环 |

从文件 I/O 角度看，`keyboard_control.py` 的输入是按键状态，输出就是这个 `FlightCommand`：

```python
FlightCommand(
    roll_rate=roll_axis * 1.0,
    pitch_rate=pitch_axis * 1.0,
    yaw_rate=yaw_axis * 1.0,
    thrust=0.6 if space_pressed else 0.0,
    exit_requested=esc_pressed,
)
```

正式自动驾驶时，应把 `KeyboardControlConsole` 替换成类似下面的命令源：

```python
class AutonomyCommandSource:
    def __init__(self, shared_data):
        self.shared_data = shared_data

    def read_command(self):
        detection = self.shared_data["vision"]["gate_detection"]
        attitude = self.shared_data["telemetry"]["attitude"]
        return FlightCommand(
            roll_rate=...,
            pitch_rate=...,
            yaw_rate=...,
            thrust=...,
            exit_requested=False,
        )
```

格式示例，数值是构造样例，不是实测飞行数据：

```python
# controller.py 输入：来自 vision_rx.py 的检测结果
shared_data["vision"]["gate_detection"] = {
    "center_px": [370, 160],
    "confidence": 0.90,
}

# 图像中心是 [320, 180]，所以 gate 偏右 50 px、偏上 20 px
error_x = 370 - 320
error_y = 160 - 180

# 一个简单比例控制器可以输出 FlightCommand
command = FlightCommand(
    roll_rate=0.0,
    pitch_rate=-0.2,   # gate 偏上，轻微调整 pitch
    yaw_rate=0.5,      # gate 偏右，向右转
    thrust=0.55,
    exit_requested=False,
)
```

这个 `FlightCommand` 随后进入 `update_attitude_flight_control()`，实际输出给模拟器的是：

```python
set_attitude_target_send(
    time_boot_ms,
    target_system,
    target_component,
    ATTITUDE_TARGET_TYPEMASK_ATTITUDE_IGNORE,
    [1, 0, 0, 0],
    0.0,   # roll_rate
    -0.2,  # pitch_rate
    0.5,   # yaw_rate
    0.55,  # thrust
)
```

控制循环频率是 `CONTROL_HZ = 60`，符合规格中控制命令频率小于 100 Hz 的限制。

| 控制方式 | 代码函数 | MAVLink 消息 / 命令 | 适合用途 | 风险 |
| -------- | -------- | ------------------- | -------- | ---- |
| 解锁无人机 | `arm()` | `MAV_CMD_COMPONENT_ARM_DISARM`，param1 = `1` | 起飞前必须执行 | 只负责 arm，不负责稳定飞行 |
| 重置模拟器 | `send_sim_reset_command()` | 自定义 command `31000` | 自动化测试、多次试跑 | 是否允许正式 timed run 使用需确认 |
| 直接电机 / actuator | `update_motor_control()` | `SET_ACTUATOR_CONTROL_TARGET` | 最底层控制、研究飞控响应 | 最难调，容易失控 |
| 姿态 / 角速度 / 推力 | `update_attitude_flight_control()` | `SET_ATTITUDE_TARGET` | 视觉伺服、竞速控制 | 需要自己设计稳定的外环 |
| 局部位置 / 速度目标 | `update_position_flight_control()` | `SET_POSITION_TARGET_LOCAL_NED` | 低速 baseline、接口实验 | 仿真器实际支持哪些字段需实测 |
| 时间同步请求 | `timesync_send()` | `TIMESYNC` | 对齐本机时间和模拟器时间 | 不是飞行控制命令 |

从动力学 I/O 角度看，当前能确认的是“输入控制量”和“可观测输出”，不能确认完整动力学方程。模拟器内部的质量、惯量、推力曲线、电机响应、空气阻力、飞控内环增益没有在 Python 模板中暴露。

可以把模拟器先当成黑箱：

```text
x(t + dt), y(t) = simulator_dynamics(x(t), u(t))
```

其中 `u(t)` 是你发送的控制量，`y(t)` 是 Python 能收到的遥测输出。已知的 I/O 关系如下：

| 控制模式 | 输入控制量 `u(t)` | 输入坐标 / 单位 | 模拟器内部含义 | 可观测输出 `y(t)` |
| ---- | ---- | ---- | ---- | ---- |
| 姿态角速度 + 推力，当前默认 | `roll_rate`、`pitch_rate`、`yaw_rate`、`thrust` | body frame；rad/s；`thrust` 为 0..1 normalized | 通过 `SET_ATTITUDE_TARGET` 交给模拟器/飞控内环；quaternion 姿态被 ignore | `ATTITUDE` 的姿态和角速度；`ODOMETRY`/`LOCAL_POSITION_NED` 的位置速度；`HIGHRES_IMU` 的加速度角速度 |
| 局部速度目标 | `vx`、`vy`、`vz` | `MAV_FRAME_LOCAL_NED`；m/s；N 前、E 右、D 下 | 通过 `SET_POSITION_TARGET_LOCAL_NED` 交给高层位置/速度控制器；当前 mask 只启用速度 | 同上，另外可用轨迹误差评估速度命令响应 |
| 直接 actuator | `controls[0:8]`，当前只用前 4 路 | actuator float，具体归一化范围未在模板说明 | 通过 `SET_ACTUATOR_CONTROL_TARGET` 直接给底层 actuator/motor 通道 | `ACTUATOR_OUTPUT_STATUS`、姿态、速度、IMU、碰撞等 |

最接近“输入什么，飞机输出什么”的当前默认关系是：

```text
输入:
FlightCommand(
    roll_rate=p_cmd,
    pitch_rate=q_cmd,
    yaw_rate=r_cmd,
    thrust=T_cmd,
)

发送:
SET_ATTITUDE_TARGET(
    body_roll_rate=p_cmd,
    body_pitch_rate=q_cmd,
    body_yaw_rate=r_cmd,
    thrust=T_cmd,
)

输出:
ATTITUDE:
    roll, pitch, yaw, rollspeed, pitchspeed, yawspeed
LOCAL_POSITION_NED / ODOMETRY:
    x, y, z, vx, vy, vz
HIGHRES_IMU:
    xacc, yacc, zacc, xgyro, ygyro, zgyro
ACTUATOR_OUTPUT_STATUS:
    actuator[0:4]
```

因此，现阶段不能写出可信的白盒方程，例如：

```text
m * a = R * thrust - m * g - drag
I * omega_dot = torque - omega x I omega
```

这些方程形式是四旋翼常见模型，但 `m`、`I`、推力系数、力矩系数、阻力和内环控制器参数在当前资料里都没有。要得到可用动力学模型，需要做系统辨识：给定一组阶跃/扫频控制输入，记录遥测输出，再拟合响应。

各控制消息的具体参数格式如下。

**解锁 arm：**

```python
command_long_send(
    target_system,
    target_component,
    MAV_CMD_COMPONENT_ARM_DISARM,
    0,  # confirmation
    1,  # param1: arm
    0, 0, 0, 0, 0, 0
)
```

如果要 disarm，通常把 `param1` 改成 `0`。

**重置模拟器：**

```python
command_long_send(
    target_system,
    target_component,
    31000,  # MAVLINK_CMD_SIM_RESET
    0,
    0, 0, 0, 0, 0, 0, 0
)
```

这个命令适合自动化测试和反复试跑；正式 timed run 中是否允许使用，需要确认规则。

**时间同步请求：**

```python
timesync_send(
    int(time.time_ns()),  # tc1: client time ns
    0                    # ts1: 0 表示 request
)
```

`timesync.py` 以 `TIMESYNC_REQUEST_HZ = 10` 的频率发送请求。当前 `setup.py` 已经调用 `TimeSync.create_timesync(...)` 启动线程。

三种主要飞行控制方式的差别：

1. **直接 actuator 控制**

   `update_motor_control()` 发送 8 路 actuator 数组。当前代码中默认数组近似是：

   ```python
   motor_rpms = [front_left, front_right, back_left, back_right, 0, 0, 0, 0]
   ```

   实际 MAVLink 调用格式是：

   ```python
   set_actuator_control_target_send(
       time_usec,
       target_system,
       target_component,
       0,          # group_mlx
       controls    # 8 个 float
   )
   ```

   当前示例常量是 `front_left=0`、`front_right=1`、`back_left=0`、`back_right=0`，只适合验证接口是否连通。这个接口需要自己处理电机混控、姿态稳定和推力分配，不适合作为第一版自动穿门方案。

2. **姿态目标 / 角速度目标控制**

   `update_attitude_flight_control()` 当前忽略 quaternion attitude，发送 body roll rate、pitch rate、yaw rate 和 thrust：

   ```text
   roll_rate, pitch_rate, yaw_rate, thrust
   ```

   实际 MAVLink 调用格式是：

   ```python
   set_attitude_target_send(
       time_boot_ms,
       target_system,
       target_component,
       ATTITUDE_TARGET_TYPEMASK_ATTITUDE_IGNORE,  # 通常为 128
       [1, 0, 0, 0],  # quaternion，被 type_mask 忽略
       roll_rate,     # rad/s
       pitch_rate,    # rad/s
       yaw_rate,      # rad/s
       thrust         # 0..1 normalized
   )
   ```

   当前 `time_boot_ms` 由 `int(time.time() * 1000) - system_boot_ms` 得到；`system_boot_ms` 是 `main.py` 启动客户端时记录的本机时间基准。严格来说，这不是模拟器 boot time，只是用于给 MAVLink 消息提供递增时间戳。

   这很适合视觉伺服。例如：

   ```text
   gate 偏左/右 → 调 yaw_rate 或 roll_rate
   gate 偏上/下 → 调 pitch_rate 或 thrust
   gate 变大/接近 → 降低前向速度或准备切换下一门
   ```

   如果仿真器对 `SET_ATTITUDE_TARGET` 响应稳定，这是后续竞速版本最值得使用的控制方式。

3. **局部位置 / 速度目标控制**

   `update_position_flight_control()` 发送 `SET_POSITION_TARGET_LOCAL_NED`，当前 type mask 忽略位置、加速度、yaw 和 yaw_rate，只使用速度：

   ```text
   frame = MAV_FRAME_LOCAL_NED
   type_mask = ignore x/y/z + ignore ax/ay/az + ignore yaw/yaw_rate
   vx = 2.0 m/s
   vy = 0.0
   vz = 0.0
   ```

   实际 MAVLink 调用格式是：

   ```python
   set_position_target_local_ned_send(
       time_boot_ms,
       target_system,
       target_component,
       MAV_FRAME_LOCAL_NED,
       VELOCITY_POSITION_MASK,  # 通常为 3527，只使用 vx/vy/vz
       0.0, 0.0, 0.0,    # x, y, z ignored
       2.0, 0.0, 0.0,    # vx, vy, vz used
       0.0, 0.0, 0.0,    # ax, ay, az ignored
       0.0,              # yaw ignored
       0.0               # yaw_rate ignored
   )
   ```

   这适合做最早期低速 baseline：先确认“发一个速度目标，飞机是否稳定前进”。之后可以扩展为：

   ```text
   根据当前 gate 的 NED 位置生成速度向量
   或根据视觉误差修正 vx / vy / vz
   ```

   但规格里强调没有 GPS / absolute global position，所以用 NED gate 真值做正式策略前必须确认是否合规。

当前代码里已经符合或已经完成的控制相关点：

1. `CONTROL_HZ = 60`，在建议的 50–80 Hz 范围内，也低于 100 Hz 限制。
2. `setup.py` 已经调用 `TimeSync.create_timesync(...)`，timesync request loop 会实际运行。
3. `main.py` 支持 `Ctrl+C` 退出，并在退出时调用 `get_thread_for_join()` 尝试停止 timesync、MAVLink RX 和 vision RX 线程。

还需要补的核心点是：把 `KeyboardControlConsole` 换成自动驾驶命令源，或者让 `Controller.update()` 直接根据 `shared_data` 计算 `FlightCommand`。

推荐第一版控制链路：

```text
vision_rx.py 检测 gate center
→ 写入 shared_data["vision"]["gate_detection"]
→ mavlink_rx.py 写入姿态、速度、active_gate_index
→ controller.py 读取 shared_data
→ 用 gate 图像中心误差计算 yaw_rate / pitch_rate / thrust
→ 通过 SET_ATTITUDE_TARGET 发送给模拟器
```

如果要先做更稳的接口验证，也可以先用 `SET_POSITION_TARGET_LOCAL_NED` 做低速速度控制，再迁移到 `SET_ATTITUDE_TARGET`。

### 3.6.6 当前模板的缺口

代码模板目前还不是完整自动驾驶系统，主要缺口如下：

1. `vision_rx.py` 的 `process_frame()` 是空函数，没有 gate 检测、图像日志或感知结果输出。
2. `mavlink_rx.py` 解析了遥测字段，但大多只是局部变量，没有写入 `shared_data`，控制器实际读不到状态。
3. `shared_data` 是普通 dict，没有锁；一旦接收线程和控制线程共享数据，应加 `threading.Lock` 或使用线程安全队列。
4. `timesync.py` 已经能发送 TIMESYNC request，但 `on_timesync()` 没有计算时钟偏移，也没有把同步结果写入 `shared_data`。
5. 当前默认控制来自 `KeyboardControlConsole`，仍然需要人工按键，不能作为 timed run 的自主 baseline。
6. `update_motor_control()` 示例常量里只有一个电机非零，如果切到 actuator 控制会很容易失控。
7. 当前没有客户端 heartbeat 发送逻辑；如果正式接口要求客户端 heartbeat 最低 2 Hz，需要补上。
8. 图像接收能处理乱序分片，但没有对丢失分片设置超时清理，长时间运行可能积累 incomplete frames。
9. `main.py` 已经有 `Ctrl+C` 和线程 join，但还没有异常保护、日志 flush、自动 reset 或批量试跑流程。
10. 没有任何数据记录、回放、自动测试或调参框架。

### 3.6.7 应该优先改哪些文件

最小可用 baseline 的改造顺序建议：

| 顺序 | 文件 | 要补的内容 |
| ---- | ---- | ---------- |
| 1 | `mavlink_rx.py` | 把 attitude、odometry、IMU、race status、track info、collision 写入 `shared_data` |
| 2 | `vision_rx.py` | 在 `process_frame()` 中做 gate 检测，并输出 bbox/center/confidence/timestamp |
| 3 | `controller.py` | 读取 `shared_data`，把键盘 `FlightCommand` 替换成 gate center servoing 或轨迹控制命令 |
| 4 | `setup.py` | 加入共享锁、logger 和 timesync offset 写入 |
| 5 | `main.py` | 增加配置、异常处理、退出清理、可选 reset 和日志路径 |

完成这五步后，文档中“通信与日志系统、Gate 检测、状态估计、控制器”的抽象模块才会和代码库真实对齐。

## 4. 最可能的高分逻辑

文档未提供正式评分标准，所以不能断言“高分=最短时间”。但从无人机竞速和 Qualification Phase 描述看，合理目标应是：

1. **合法完成赛道**；
2. **不人工干预**；
3. **不漏门、不撞门、不出界**；
4. **在稳定完成基础上尽可能缩短用时**。

因此策略排序应为：

| 优先级 | 目标        | 原因                                       |
| --: | --------- | ---------------------------------------- |
|   1 | 100% 自动完成 | 不完成或人工干预没有分数意义                           |
|   2 | 稳定穿门      | gate 是核心任务，碰撞或漏门风险最大                     |
|   3 | 降低视觉丢失率   | 高速时 gate 离开视野会直接失败                       |
|   4 | 优化路径和速度   | 只有稳定后才值得追求快                              |
|   5 | 做赛道确定性优化  | 环境 deterministic，可通过反复测试找到更快 racing line |

## 5. 推荐技术路线

### 路线 A：最快可落地的可靠版本

适合先拿到有效成绩。

架构：

```text
UDP 图像重组
→ gate 检测
→ gate 中心偏差估计
→ 视觉伺服控制
→ 速度 / yaw / pitch / roll 命令
→ 穿过 gate 后寻找下一个 gate
```

核心做法：

1. 使用传统视觉或轻量模型检测 gate。

   * 如果 gate 颜色稳定，可用 HSV 阈值、边缘、轮廓、矩形检测。
   * 如果光照变化明显，训练一个轻量 YOLO / segmentation 模型。
2. 以 gate 内孔中心作为目标。
3. 图像中 gate 中心偏左，则 yaw/roll 修正；偏上/下，则 pitch/thrust 修正。
4. gate 尺寸变大说明距离变近，可用 apparent size 粗略估距。
5. 先低速穿门，保证完成。
6. 记录每次失败时的图像、姿态、速度和命令。

优点：开发快，不依赖完整 3D 地图。
缺点：高速时稳定性有限，转弯和遮挡场景容易失败。

### 路线 B：更适合高分的版本

适合在可靠完成后提速。

架构：

```text
图像 + 遥测
→ gate 角点检测 / segmentation
→ PnP 相对位姿估计
→ EKF / 状态估计
→ gate-to-gate 轨迹规划
→ 速度曲线优化
→ MPC / cascaded PID 控制
```

核心做法：

1. 检测 gate 的四个内孔角点或外框角点。
2. 利用已知 gate 尺寸和相机内参做 PnP，估计 gate 相对相机的 6DoF 位姿。
3. 加入相机相对机体向上 20° 的外参修正。
4. 用 IMU、姿态、速度做短时预测，在两帧图像之间补偿 30 Hz 视觉延迟。
5. 每个 gate 设置三个 waypoint：

   * pre-gate approach point；
   * gate center；
   * post-gate exit point。
6. 根据 gate 角度和下一个 gate 方向生成 racing line。
7. 对直线段提速，对转弯段提前减速。
8. 对视觉置信度低、gate 边缘不完整、距离过近时自动进入保守模式。

这条路线更有机会拿高分，因为它不是“看到门就飞过去”，而是提前规划姿态、速度和下一扇门的进入角。

## 6. 可以利用的比赛特性

### 6.1 赛道 deterministic

文档明确说 course geometry、physics parameters、environmental conditions 都是 deterministic。

可以做：

1. 反复跑同一赛道，记录最优轨迹；
2. 建立 gate 顺序和相对位姿表；
3. 用自动化脚本扫速度参数；
4. 用 Bayesian optimization / CMA-ES / grid search 优化每段速度；
5. 保存每个 gate 的最佳 approach angle；
6. 对特定弯道设计专门策略。

但要确认主办方是否允许使用预先记录的赛道信息。文档没有禁止，但也没有明确授权。

### 6.2 Gate 尺寸已知

已知 gate 内孔 1.5 m × 1.5 m，可以用于：

1. PnP 估计距离；
2. 判断是否正对 gate；
3. 估计穿门时间；
4. 设置速度上限；
5. 计算安全 corridor。

### 6.3 相机无畸变

文档说是标准 pinhole camera model，无 lens distortion。

这降低了视觉定位难度。可以直接用 OpenCV 的 solvePnP、homography、角点重投影误差做定位和置信度判断。

## 7. 最重要的工程模块

### 7.1 通信与日志系统

这是第一优先级，不是视觉模型。

必须实现：

1. MAVLink 连接；
2. heartbeat 维护；
3. command stream；
4. telemetry parser；
5. TIMESYNC 处理；
6. vision UDP packet reassembly；
7. 全量日志记录。

建议日志字段：

| 类型 | 字段                                           |
| -- | -------------------------------------------- |
| 时间 | sim_time_ns、本机时间、延迟                          |
| 图像 | frame_id、是否完整、JPEG size                      |
| 遥测 | attitude、IMU、velocity、system status          |
| 感知 | gate bbox、角点、置信度、相对位姿                        |
| 规划 | 当前目标 gate、waypoint、目标速度                      |
| 控制 | 发送的 attitude / position target / thrust      |
| 事件 | gate passed、collision、lost tracking、recovery |

没有日志就无法提速。

### 7.2 Gate 检测

建议分阶段：

第一版：传统 CV。

* 色彩阈值；
* 边缘检测；
* 轮廓筛选；
* 矩形/方框检测；
* gate center servoing。

第二版：深度学习。

* YOLO 检测 gate；
* segmentation 分割 gate 框；
* corner/keypoint model 预测四个角点；
* 加入动态光照和模糊的数据增强。

高分版本更推荐 keypoint/segmentation，而不是只做 bbox。bbox 只能粗略对准，角点可以做真实位姿估计。

### 7.3 状态估计

由于没有 GPS 和全局位置，状态估计应以“相对 gate 位姿”为核心。

建议：

```text
IMU / attitude / velocity prediction
+ camera-based gate pose update
+ confidence-based measurement weighting
```

如果视觉置信度高，用 gate pose 修正；如果短暂丢失，用 IMU 和速度预测维持 0.5–1 秒；超过阈值则减速、搜索 gate。

### 7.4 控制器

可以做两层：

1. 外层：轨迹和速度规划；
2. 内层：姿态 / 速度 / yaw 控制。

如果仿真器对 `SET_ATTITUDE_TARGET` 支持稳定，竞速性能通常会比高层位置命令更好；如果时间有限，先用 `SET_POSITION_TARGET_LOCAL_NED` 或速度目标做原型，再迁移到姿态目标。

控制目标：

1. 穿门时横向误差小；
2. yaw 对准 gate normal；
3. 不在 gate 平面附近大幅横滚；
4. 过门后快速转向下一门；
5. 转弯前减速，出弯加速。

## 8. 高分策略：先稳后快

### 阶段 1：拿到能完成的 baseline

目标：低速、不撞、能穿完。

参数建议：

| 项目   | 建议                         |
| ---- | -------------------------- |
| 速度   | 保守固定速度                     |
| 控制频率 | 50–80 Hz                   |
| 视觉   | 30 Hz gate center servoing |
| 安全策略 | gate 丢失就减速                 |
| 穿门策略 | 通过 gate 后继续前飞一小段再找下一门      |

### 阶段 2：提高成功率

目标：多次重复运行成功率接近 100%。

要做：

1. gate 检测置信度；
2. 丢帧处理；
3. 视觉延迟补偿；
4. 转弯前速度限制；
5. 异常恢复逻辑；
6. 自动化回归测试。

建议每次改动后跑至少 20 次，统计：

| 指标        | 目标         |
| --------- | ---------- |
| 完赛率       | >95%       |
| 撞门次数      | 接近 0       |
| gate 丢失次数 | 越低越好       |
| 平均横向穿门误差  | <0.3–0.4 m |
| 最大横向穿门误差  | <0.5 m     |
| 控制延迟      | 稳定、可预测     |

### 阶段 3：优化速度

目标：在不显著降低完赛率的前提下缩短时间。

可优化：

1. 每段最大速度；
2. 每个 gate 的 approach angle；
3. 过门后是否提前转向；
4. 直线段加速度；
5. 转弯段减速点；
6. gate 附近 yaw alignment；
7. 视觉置信度低时的速度降级策略。

### 阶段 4：赛道特化优化

因为环境 deterministic，可以为每个 gate 做专门参数：

```text
gate 1: 起步加速
gate 2: 保守通过
gate 3: 提前右转
gate 4: 减速进门
...
```

更进一步，可以自动搜索每段参数：

```text
参数 = [每段目标速度, pre-gate distance, post-gate distance, yaw gain, lateral gain]
目标 = 完赛时间 + 碰撞惩罚 + 漏门惩罚 + 横向误差惩罚
优化 = Bayesian optimization / CMA-ES / grid search
```

## 12. 结论

从这份规格看，拿高分的关键不是单点算法，而是完整系统工程：

1. **先做可靠完赛**，否则速度没有意义；
2. **充分利用 gate 尺寸、相机内参和 deterministic 赛道**；
3. **用视觉相对定位替代 GPS/全局位置依赖**；
4. **控制频率小于 100 Hz，但控制层要比视觉层更高频**；
5. **把日志、自动测试和调参系统作为核心模块**；
6. **后期通过赛道特化 racing line 和速度曲线拿时间优势**。

最推荐路线是：先用传统视觉 + 视觉伺服做 baseline，然后升级到 gate keypoint/PnP + 状态估计 + 轨迹规划 + 自动调参。

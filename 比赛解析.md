根据上传的 AI Grand Prix Virtual Qualifier 技术规格文件，比赛核心不是“写一个 AI 模型”本身，而是做一个**无人机自主飞行系统**：用前视相机和 MAVLink 遥测，在仿真环境中自动穿过起点门、中间门和终点门，并且提交计时飞行时不能有人为干预。

## 1. 比赛目标

Round One 的目标是验证参赛软件能否成功导航完整赛道。赛道结构包括：

| 项目   | 内容                                         |
| ---- | ------------------------------------------ |
| 起点   | start gate                                 |
| 中间赛道 | intermediate gates / sequential race gates |
| 终点   | finish gate                                |
| 最长时间 | 8 分钟                                       |
| 提交要求 | 计时飞行过程中禁止人为干预，否则直接取消资格                     |

文档没有给出明确评分公式，比如是否按总时间排名、是否有碰撞惩罚、漏门惩罚、重置惩罚等。因此目前只能确定：**必须自主完成赛道；高分策略应优先保证稳定完成，再优化速度**。

## 2. 硬性限制

### 2.1 不能人工接管

提交的 timed run 飞行过程中，任何 human interaction 都会导致 immediate disqualification。这意味着所有感知、规划、控制、纠错都必须在软件中自动完成。

### 2.2 运行环境限制

文档说明仿真器运行在 Windows 11，标准 PC，GPU 显存约 8GB；当前不支持 Linux。参赛者可以使用 Python，文档提到 Python 3.14.2 可正常运行，也允许其他环境。

工程含义：不要把主开发环境只放在 Linux 上。需要尽早在 Windows 11 上验证依赖、网络、MAVLink、图像流解码和控制频率。

### 2.3 控制频率限制

仿真物理频率是 120 Hz，但控制命令频率必须小于 100 Hz，heartbeat 最低 2 Hz。

建议控制频率设置在 **50–80 Hz**，视觉处理 30 Hz，控制层在两帧图像之间做状态预测，不要每收到图像才发一次控制。

### 2.4 没有 GPS 和全局位置

仿真器内部使用局部笛卡尔坐标，但文档明确说没有 GPS 仿真，也不暴露 absolute global position。

这很关键。不能依赖“知道自己在赛道全局坐标中的位置”。系统需要主要依赖：

1. 前视相机识别 gate；
2. IMU、姿态、速度等遥测；
3. 视觉相对定位；
4. 必要时用已知赛道的确定性做前馈路径。

## 3. 提供的技术资料

### 3.1 仿真环境

仿真环境包括起点门、顺序赛门、终点门、垂直和水平障碍、边界、地形与环境结构。赛道几何、物理参数和环境条件对所有参赛者一致，并且是 deterministic。

这意味着可以做大量重复测试，利用同一赛道的确定性调参、优化轨迹、记录失败案例。

### 3.2 飞行器尺寸

无人机尺寸：

| 尺寸 |     数值 |
| -- | -----: |
| 宽  | 280 mm |
| 长  | 280 mm |
| 高  | 160 mm |

Gate 外边界为 2700 × 2700 × 260 mm，内孔为 1500 × 1500 × 260 mm。

由此可估算安全余量：

| 方向 |    内孔尺寸 |   机体尺寸 |   单侧理论余量 |
| -- | ------: | -----: | -------: |
| 横向 | 1500 mm | 280 mm | 约 610 mm |
| 垂直 | 1500 mm | 160 mm | 约 670 mm |

理论余量不小，但高速飞行时姿态倾斜、控制延迟、视觉误差、碰撞模型都会吃掉余量。实际策略不应贴边飞，建议让 gate center 误差控制在 **0.3–0.4 m 以内**，高速段也尽量不要超过 0.5 m。

### 3.3 坐标系与相机

MAVLink 坐标 convention 是 NED：

| Frame               | 含义                      |
| ------------------- | ----------------------- |
| MAV_FRAME_LOCAL_NED | 原点为地面固定点，通常是无人机 arm 的位置 |
| MAV_FRAME_BODY_NED  | 原点为机体，X 向前，Y 向右，Z 向下    |

相机和机体同原点，相机相对机体向上倾斜 20°。IMU 到机体是 identity map。相机是 pinhole model，无畸变，分辨率 640 × 360，主点 [320, 180]，fx = fy = 320。

注意：文档写了 “VFoV = 90°”，但 fx = fy = 320 且分辨率 640 × 360 时，标准 pinhole 计算得到 horizontal FOV 约 90°，vertical FOV 约 58.7°。这里可能存在“VFoV/HFoV 标注不一致”。实现投影、PnP、视觉伺服前，必须用仿真图像实测验证内参，不要盲信单一字段。

### 3.4 通信接口

通信通过 MAVLink 2 / MAVSDK-compatible interface，传输使用 UDP。支持消息包括：

| 消息                            | 方向                 | 用途        |
| ----------------------------- | ------------------ | --------- |
| HEARTBEAT                     | Simulator → Client | 连接状态      |
| ATTITUDE                      | Simulator → Client | 姿态        |
| HIGHRES_IMU                   | Simulator → Client | 车辆状态 / 测量 |
| SET_POSITION_TARGET_LOCAL_NED | Client → Simulator | 控制接口      |
| SET_ATTITUDE_TARGET           | Client → Simulator | 控制接口      |
| TIMESYNC                      | Simulator → Client | 时间同步      |



文档没有说明 `SET_POSITION_TARGET_LOCAL_NED` 中哪些字段或 type mask 被仿真器实际使用，也没有说明 `SET_ATTITUDE_TARGET` 的 thrust/attitude/yaw-rate 具体语义。因此开发时要尽早做接口实验：分别发送速度、位置、加速度、姿态、推力命令，记录无人机响应，确认真实控制通道。

### 3.5 图像流

相机图像流频率为 30 Hz，分辨率 640 × 360，UDP 默认端口 5600。每张 JPEG 图像被拆成多个 packet，每个 packet 有 24 字节 metadata header，包含 frame_id、chunk_id、total_chunks、jpeg_size、payload_size 和 sim_time_ns。

工程重点：

1. 必须按 frame_id 重组 packet；
2. 必须能处理乱序、丢包、重复包；
3. incomplete frame 要丢弃，不要把坏图送进感知模型；
4. 用 sim_time_ns 对齐图像和遥测；
5. 记录图像延迟，否则高速飞行会明显偏门。

## 3.6 当前代码库对照

当前文件夹不是完整源码工程，而是一个**打包好的 Windows 仿真器**加一个**Python 自动驾驶客户端模板**。仿真器内部 Unreal / FlightSim 逻辑已经被打包，主要可改部分在 `PyAIPilotExample`。

| 路径 | 作用 | 是否主要修改 |
| ---- | ---- | ------------ |
| `AIGP_3364/FlightSim.exe` | 仿真器启动入口 | 否 |
| `AIGP_3364/FlightSim/Binaries/Win64/DCGame-Win64-Shipping.exe` | Unreal 打包后的实际运行程序 | 否 |
| `AIGP_3364/FlightSim/Content/Paks/FlightSim-WindowsNoEditor.pak` | 赛道、模型、材质等打包资源 | 通常否 |
| `AIGP_3364/FlightSim/Content/FMOD/Desktop/*.bank` | FMOD 音频资源 | 否 |
| `AIGP_3364/Engine/*` | Unreal Engine 运行时依赖 | 否 |
| `PyAIPilotExample/main.py` | Python 客户端入口，设置 MAVLink 地址端口，创建组件，arm 后进入控制循环 | 是 |
| `PyAIPilotExample/setup.py` | 建立 MAVLink 连接，创建 MAVLink 接收器、图像接收器和控制器 | 是 |
| `PyAIPilotExample/mavlink_rx.py` | 接收并解析 MAVLink 遥测、比赛状态、赛道 gate 信息、碰撞信息 | 是 |
| `PyAIPilotExample/vision_rx.py` | 接收 UDP 图像分片，重组 JPEG，用 OpenCV 解码成图像 | 是 |
| `PyAIPilotExample/controller.py` | 发送 arm/reset/电机/姿态/位置控制命令 | 是 |
| `PyAIPilotExample/timesync.py` | TIMESYNC 请求循环定义 | 是 |
| `PyAIPilotExample/requirements.txt` | Python 依赖：pymavlink、opencv-python、numpy、matplotlib、keyboard | 视情况 |

### 3.6.1 启动和连接流程

当前模板的实际启动链路是：

```text
启动 AIGP_3364/FlightSim.exe
→ 进入仿真器并让它开始发送 MAVLink / vision 数据
→ python PyAIPilotExample/main.py
→ setup_components(...)
→ mavutil.mavlink_connection("udpin:127.0.0.1:14550")
→ wait_heartbeat()
→ 启动 MAVLinkRX 接收线程
→ 创建 TimeSync 对象
→ 启动 VisionRX 图像接收线程
→ 创建 Controller
→ controller.arm()
→ while True: controller.update()
```

也就是说，Python 客户端默认在本机监听 MAVLink UDP `127.0.0.1:14550`，图像接收器默认绑定 `0.0.0.0:5600`。如果仿真器和 Python 不在同一台机器，需要同时修改 `main.py` 里的 MAVLink 地址，以及确认仿真器的 UDP 输出目标。

### 3.6.2 代码中的实际数据流

MAVLink 状态流：

```text
Simulator
→ UDP 14550
→ setup.py / mavutil.mavlink_connection(...)
→ mavlink_rx.py / recv_match(blocking=False)
→ on_heartbeat / on_attitude / on_odometry / on_highres_imu / on_collision / ...
→ shared_data 或日志系统
→ controller.py
```

图像流：

```text
Simulator
→ UDP 5600
→ vision_rx.py / sock.recvfrom(...)
→ 24 字节 header: frame_id, chunk_id, total_chunks, jpeg_size, payload_size, sim_time_ns
→ 按 frame_id 和 chunk_id 重组 JPEG
→ cv2.imdecode(...)
→ process_frame(frame_id, image)
→ gate 检测 / 位姿估计
→ shared_data 或日志系统
→ controller.py
```

控制命令流：

```text
controller.py
→ arm: MAV_CMD_COMPONENT_ARM_DISARM
→ reset: 自定义 MAVLINK_CMD_SIM_RESET = 31000
→ set_actuator_control_target_send(...)
→ set_attitude_target_send(...)
→ set_position_target_local_ned_send(...)
→ Simulator
```

代码里已经提供了三类控制接口：

| 函数 | MAVLink 消息 | 当前含义 |
| ---- | ------------ | -------- |
| `update_motor_control()` | `SET_ACTUATOR_CONTROL_TARGET` | 直接发送 8 路 actuator/motor 数组 |
| `update_attitude_flight_control()` | `SET_ATTITUDE_TARGET` | 发送 roll/pitch/yaw rate 和 thrust |
| `update_position_flight_control()` | `SET_POSITION_TARGET_LOCAL_NED` | 示例发送 NED 速度目标 |

当前 `Controller.update()` 默认调用的是 `update_motor_control()`，不是视觉伺服、姿态控制或轨迹控制。因此它只是接口样例，不是能完成比赛的 baseline。

### 3.6.3 MAVLink 解析内容和可利用信息

`mavlink_rx.py` 目前能识别这些消息类型：

| 消息 | 代码处理函数 | 可用信息 |
| ---- | ------------ | -------- |
| `HEARTBEAT` | `on_heartbeat()` | 是否 armed |
| `TIMESYNC` | `on_timesync()` | 请求/响应时间戳 |
| `ATTITUDE` | `on_attitude()` | roll、pitch、yaw、角速度 |
| `LOCAL_POSITION_NED` | `on_local_position_ned()` | NED 位置和速度 |
| `ODOMETRY` | `on_odometry()` | 位置、四元数、速度、角速度、reset_count |
| `HIGHRES_IMU` | `on_highres_imu()` | 加速度和陀螺仪 |
| `ACTUATOR_OUTPUT_STATUS` | `on_actuator_output_status()` | 4 个电机输出 |
| `COLLISION` | `on_collision()` | gate / environment 碰撞 ID、威胁等级、冲量 |
| `ENCAPSULATED_DATA` | `on_encapsulated_data()` | race status 或 track data |
| `DATA_TRANSMISSION_HANDSHAKE` | 接收循环内处理 | track data 分片数量 |

其中 `ENCAPSULATED_DATA` 有两个自定义 payload：

| data_type | 含义 | 字段 |
| --------- | ---- | ---- |
| `1` | race status | sim_boot_time_ms、race_start_boot_time_ms、race_finish_time_ns、active_gate_index、last_gate_race_time |
| `2` | track info | gate_id、gate 在 NED 坐标中的位置、四元数朝向、width、height |

这说明代码模板实际上暴露了比技术规格摘要更多的信息：不仅有相机和基础遥测，还有当前 active gate index、赛道 gate 几何、碰撞事件和 actuator output。这些信息能显著降低开发难度，但是否允许在正式提交中依赖 track info 做赛道特化，需要再确认主办方规则。

### 3.6.4 当前模板的缺口

代码模板目前还不是完整自动驾驶系统，主要缺口如下：

1. `vision_rx.py` 的 `process_frame()` 是空函数，没有 gate 检测、图像日志或感知结果输出。
2. `mavlink_rx.py` 解析了遥测字段，但大多只是局部变量，没有写入 `shared_data`，控制器实际读不到状态。
3. `shared_data` 是普通 dict，没有锁；一旦接收线程和控制线程共享数据，应加 `threading.Lock` 或使用线程安全队列。
4. `timesync.py` 定义了 `TimeSync.create_timesync()` 和 `timesync_loop()`，但 `setup.py` 里只是 `TimeSync(sim_conn, shared_data)`，没有真正启动 timesync 线程。
5. `controller.py` 的 `CONTROL_HZ = 250`，高于规格中“控制命令频率必须小于 100 Hz”的要求，应改到 50–80 Hz 区间。
6. 当前默认控制是固定电机数组，而且示例常量里只有一个电机非零，不能作为飞行控制 baseline。
7. 当前没有客户端 heartbeat 发送逻辑；如果正式接口要求客户端 heartbeat 最低 2 Hz，需要补上。
8. 图像接收能处理乱序分片，但没有对丢失分片设置超时清理，长时间运行可能积累 incomplete frames。
9. `main.py` 是无限循环，没有 Ctrl+C 之外的优雅退出、异常保护、日志 flush 或自动 reset 流程。
10. 没有任何数据记录、回放、自动测试或调参框架。

### 3.6.5 应该优先改哪些文件

最小可用 baseline 的改造顺序建议：

| 顺序 | 文件 | 要补的内容 |
| ---- | ---- | ---------- |
| 1 | `mavlink_rx.py` | 把 attitude、odometry、IMU、race status、track info、collision 写入 `shared_data` |
| 2 | `vision_rx.py` | 在 `process_frame()` 中做 gate 检测，并输出 bbox/center/confidence/timestamp |
| 3 | `controller.py` | 降低 `CONTROL_HZ`，读取 `shared_data`，从固定命令改成 gate center servoing |
| 4 | `setup.py` | 启动真正的 timesync 线程，加入共享锁和 logger |
| 5 | `main.py` | 增加配置、异常处理、退出清理、可选 reset 和日志路径 |

完成这五步后，文档中“通信与日志系统、Gate 检测、状态估计、控制器”的抽象模块才会和代码库真实对齐。

## 4. 最可能的高分逻辑

文档未提供正式评分标准，所以不能断言“高分=最短时间”。但从无人机竞速和 Qualification Phase 描述看，合理目标应是：

1. **合法完成赛道**；
2. **不人工干预**；
3. **不漏门、不撞门、不出界**；
4. **在稳定完成基础上尽可能缩短用时**。

因此策略排序应为：

| 优先级 | 目标        | 原因                                       |
| --: | --------- | ---------------------------------------- |
|   1 | 100% 自动完成 | 不完成或人工干预没有分数意义                           |
|   2 | 稳定穿门      | gate 是核心任务，碰撞或漏门风险最大                     |
|   3 | 降低视觉丢失率   | 高速时 gate 离开视野会直接失败                       |
|   4 | 优化路径和速度   | 只有稳定后才值得追求快                              |
|   5 | 做赛道确定性优化  | 环境 deterministic，可通过反复测试找到更快 racing line |

## 5. 推荐技术路线

### 路线 A：最快可落地的可靠版本

适合先拿到有效成绩。

架构：

```text
UDP 图像重组
→ gate 检测
→ gate 中心偏差估计
→ 视觉伺服控制
→ 速度 / yaw / pitch / roll 命令
→ 穿过 gate 后寻找下一个 gate
```

核心做法：

1. 使用传统视觉或轻量模型检测 gate。

   * 如果 gate 颜色稳定，可用 HSV 阈值、边缘、轮廓、矩形检测。
   * 如果光照变化明显，训练一个轻量 YOLO / segmentation 模型。
2. 以 gate 内孔中心作为目标。
3. 图像中 gate 中心偏左，则 yaw/roll 修正；偏上/下，则 pitch/thrust 修正。
4. gate 尺寸变大说明距离变近，可用 apparent size 粗略估距。
5. 先低速穿门，保证完成。
6. 记录每次失败时的图像、姿态、速度和命令。

优点：开发快，不依赖完整 3D 地图。
缺点：高速时稳定性有限，转弯和遮挡场景容易失败。

### 路线 B：更适合高分的版本

适合在可靠完成后提速。

架构：

```text
图像 + 遥测
→ gate 角点检测 / segmentation
→ PnP 相对位姿估计
→ EKF / 状态估计
→ gate-to-gate 轨迹规划
→ 速度曲线优化
→ MPC / cascaded PID 控制
```

核心做法：

1. 检测 gate 的四个内孔角点或外框角点。
2. 利用已知 gate 尺寸和相机内参做 PnP，估计 gate 相对相机的 6DoF 位姿。
3. 加入相机相对机体向上 20° 的外参修正。
4. 用 IMU、姿态、速度做短时预测，在两帧图像之间补偿 30 Hz 视觉延迟。
5. 每个 gate 设置三个 waypoint：

   * pre-gate approach point；
   * gate center；
   * post-gate exit point。
6. 根据 gate 角度和下一个 gate 方向生成 racing line。
7. 对直线段提速，对转弯段提前减速。
8. 对视觉置信度低、gate 边缘不完整、距离过近时自动进入保守模式。

这条路线更有机会拿高分，因为它不是“看到门就飞过去”，而是提前规划姿态、速度和下一扇门的进入角。

## 6. 可以利用的比赛特性

### 6.1 赛道 deterministic

文档明确说 course geometry、physics parameters、environmental conditions 都是 deterministic。

可以做：

1. 反复跑同一赛道，记录最优轨迹；
2. 建立 gate 顺序和相对位姿表；
3. 用自动化脚本扫速度参数；
4. 用 Bayesian optimization / CMA-ES / grid search 优化每段速度；
5. 保存每个 gate 的最佳 approach angle；
6. 对特定弯道设计专门策略。

但要确认主办方是否允许使用预先记录的赛道信息。文档没有禁止，但也没有明确授权。

### 6.2 Gate 尺寸已知

已知 gate 内孔 1.5 m × 1.5 m，可以用于：

1. PnP 估计距离；
2. 判断是否正对 gate；
3. 估计穿门时间；
4. 设置速度上限；
5. 计算安全 corridor。

### 6.3 相机无畸变

文档说是标准 pinhole camera model，无 lens distortion。

这降低了视觉定位难度。可以直接用 OpenCV 的 solvePnP、homography、角点重投影误差做定位和置信度判断。

## 7. 最重要的工程模块

### 7.1 通信与日志系统

这是第一优先级，不是视觉模型。

必须实现：

1. MAVLink 连接；
2. heartbeat 维护；
3. command stream；
4. telemetry parser；
5. TIMESYNC 处理；
6. vision UDP packet reassembly；
7. 全量日志记录。

建议日志字段：

| 类型 | 字段                                           |
| -- | -------------------------------------------- |
| 时间 | sim_time_ns、本机时间、延迟                          |
| 图像 | frame_id、是否完整、JPEG size                      |
| 遥测 | attitude、IMU、velocity、system status          |
| 感知 | gate bbox、角点、置信度、相对位姿                        |
| 规划 | 当前目标 gate、waypoint、目标速度                      |
| 控制 | 发送的 attitude / position target / thrust      |
| 事件 | gate passed、collision、lost tracking、recovery |

没有日志就无法提速。

### 7.2 Gate 检测

建议分阶段：

第一版：传统 CV。

* 色彩阈值；
* 边缘检测；
* 轮廓筛选；
* 矩形/方框检测；
* gate center servoing。

第二版：深度学习。

* YOLO 检测 gate；
* segmentation 分割 gate 框；
* corner/keypoint model 预测四个角点；
* 加入动态光照和模糊的数据增强。

高分版本更推荐 keypoint/segmentation，而不是只做 bbox。bbox 只能粗略对准，角点可以做真实位姿估计。

### 7.3 状态估计

由于没有 GPS 和全局位置，状态估计应以“相对 gate 位姿”为核心。

建议：

```text
IMU / attitude / velocity prediction
+ camera-based gate pose update
+ confidence-based measurement weighting
```

如果视觉置信度高，用 gate pose 修正；如果短暂丢失，用 IMU 和速度预测维持 0.5–1 秒；超过阈值则减速、搜索 gate。

### 7.4 控制器

可以做两层：

1. 外层：轨迹和速度规划；
2. 内层：姿态 / 速度 / yaw 控制。

如果仿真器对 `SET_ATTITUDE_TARGET` 支持稳定，竞速性能通常会比高层位置命令更好；如果时间有限，先用 `SET_POSITION_TARGET_LOCAL_NED` 或速度目标做原型，再迁移到姿态目标。

控制目标：

1. 穿门时横向误差小；
2. yaw 对准 gate normal；
3. 不在 gate 平面附近大幅横滚；
4. 过门后快速转向下一门；
5. 转弯前减速，出弯加速。

## 8. 高分策略：先稳后快

### 阶段 1：拿到能完成的 baseline

目标：低速、不撞、能穿完。

参数建议：

| 项目   | 建议                         |
| ---- | -------------------------- |
| 速度   | 保守固定速度                     |
| 控制频率 | 50–80 Hz                   |
| 视觉   | 30 Hz gate center servoing |
| 安全策略 | gate 丢失就减速                 |
| 穿门策略 | 通过 gate 后继续前飞一小段再找下一门      |

### 阶段 2：提高成功率

目标：多次重复运行成功率接近 100%。

要做：

1. gate 检测置信度；
2. 丢帧处理；
3. 视觉延迟补偿；
4. 转弯前速度限制；
5. 异常恢复逻辑；
6. 自动化回归测试。

建议每次改动后跑至少 20 次，统计：

| 指标        | 目标         |
| --------- | ---------- |
| 完赛率       | >95%       |
| 撞门次数      | 接近 0       |
| gate 丢失次数 | 越低越好       |
| 平均横向穿门误差  | <0.3–0.4 m |
| 最大横向穿门误差  | <0.5 m     |
| 控制延迟      | 稳定、可预测     |

### 阶段 3：优化速度

目标：在不显著降低完赛率的前提下缩短时间。

可优化：

1. 每段最大速度；
2. 每个 gate 的 approach angle；
3. 过门后是否提前转向；
4. 直线段加速度；
5. 转弯段减速点；
6. gate 附近 yaw alignment；
7. 视觉置信度低时的速度降级策略。

### 阶段 4：赛道特化优化

因为环境 deterministic，可以为每个 gate 做专门参数：

```text
gate 1: 起步加速
gate 2: 保守通过
gate 3: 提前右转
gate 4: 减速进门
...
```

更进一步，可以自动搜索每段参数：

```text
参数 = [每段目标速度, pre-gate distance, post-gate distance, yaw gain, lateral gain]
目标 = 完赛时间 + 碰撞惩罚 + 漏门惩罚 + 横向误差惩罚
优化 = Bayesian optimization / CMA-ES / grid search
```

## 12. 结论

从这份规格看，拿高分的关键不是单点算法，而是完整系统工程：

1. **先做可靠完赛**，否则速度没有意义；
2. **充分利用 gate 尺寸、相机内参和 deterministic 赛道**；
3. **用视觉相对定位替代 GPS/全局位置依赖**；
4. **控制频率小于 100 Hz，但控制层要比视觉层更高频**；
5. **把日志、自动测试和调参系统作为核心模块**；
6. **后期通过赛道特化 racing line 和速度曲线拿时间优势**。

最推荐路线是：先用传统视觉 + 视觉伺服做 baseline，然后升级到 gate keypoint/PnP + 状态估计 + 轨迹规划 + 自动调参。
